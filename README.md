# Zibll 商城迁移助手

用于在两个 WordPress 网站之间安全迁移 Zibll 商城商品的独立插件。新版重写了原“全量克隆版”的导入导出逻辑，重点解决 Meta 丢失、附件 ID 错乱、正文图片未迁移、失败误报成功和不安全反序列化等问题。

作者：[Mao TK](https://www.maotk.com/)

## 下载

最新可安装版本：

[`dist/zibll-shop-migrator-6.1.3.zip`](dist/zibll-shop-migrator-6.1.3.zip)

## 主要功能

- 逐条选择需要迁移的商品。
- 按已发布、私密、草稿、待审和定时状态筛选。
- 搜索商品标题、别名或 ID。
- 支持全选、取消全选和选择当前搜索结果。
- 迁移商品标题、正文、摘要、别名、日期、状态和密码。
- 迁移完整商品 Meta，包括价格、库存、销量、规格、属性和购买设置。
- 保留同一 Meta 字段的全部值，不再只导入第一个值。
- 迁移商品分类与标签的名称、别名、描述和父子层级。
- 迁移特色图、正文图片、响应式图片、懒加载图片和 CSS 背景图。
- 识别图片类 Meta 中保存的附件 ID，并替换为新站附件 ID。
- 图片打包进 ZIP，新站不再依赖旧站长期在线。
- 图片按内容哈希去重，重复导入不会反复创建同一媒体。
- 支持覆盖之前导入或商品别名相同的商品。
- 显示导出/导入进度、耗时和导出完成提示。
- 分别报告失败商品、失败图片和其他警告。

## 相比旧版修复了什么

### 1. Meta 多值丢失

旧版只读取每个 Meta 字段的第一个值：

```php
$values[0]
```

这会导致多值属性、图库、规则或历史记录丢失。新版会删除目标商品对应字段后，按原顺序逐个恢复全部值。

### 2. 旧附件 ID 污染新站

旧版会原样复制 `_thumbnail_id` 和其他附件 ID。相同数字在新站可能指向完全不同的附件。

新版：

- 排除 `_thumbnail_id` 等系统字段。
- 单独迁移特色图片。
- 检测图片相关 Meta 中的旧附件 ID。
- 导入后替换为新站实际生成的附件 ID。

### 3. 正文图片没有迁移

旧版只下载特色图，虽然导出了来源域名，但没有替换正文图片。

新版会处理：

```text
src
srcset
data-src
data-original
data-lazy-src
CSS url(...)
常见图片直链
```

导入成功后，商品正文会改用新站媒体库地址。

### 4. 失败误报成功

旧版没有让 `wp_insert_post()` 返回 `WP_Error`，商品写入失败时可能显示“成功 ID: 0”。

新版使用 WordPress 错误模式并检查有效商品 ID。失败商品会显示标题和具体原因。

### 5. 不安全反序列化

旧版直接对导入内容执行 `maybe_unserialize()`，恶意数据可能触发 PHP 对象注入。

新版：

- 只在确认是序列化数据时解码。
- 使用 `allowed_classes => false`。
- 检测并拒绝所有对象。
- 序列化数组和普通标量仍可正常迁移。

### 6. 标题查重误判

旧版仅按标题判断重复。同名商品会被错误跳过，改名商品又可能重复创建。

新版优先使用：

```text
来源网站 + 原商品 ID
```

找不到时再使用商品别名作为后备匹配。

### 7. 分类层级丢失

旧版只保存分类名称。新版同时保存分类别名、描述、父分类和实际关联状态。

### 8. 图片失败无提示

新版报告会列出：

- 关联商品名称
- 原图片地址
- 下载、格式识别或上传失败原因

### 9. 固定作者账号

旧版会优先把商品归到用户名 `micdu`。新版商品归当前执行导入的管理员，不再依赖固定用户名。

### 10. 全局修改 PHP 环境

旧版在每次 WordPress 请求中执行无限时间和固定 512M 内存。新版只在管理员真正开始导入或导出时申请更高资源。

## 安装

1. 下载 `dist` 目录中的 ZIP。
2. 登录 WordPress 后台。
3. 打开“插件 → 安装插件 → 上传插件”。
4. 上传 ZIP 并安装。
5. 启用“Zibll 商城迁移助手”。
6. 打开“工具 → Zibll 商城迁移”。

也可以把仓库中的 `zibll-shop-migrator` 目录上传到：

```text
wp-content/plugins/
```

## 完整迁移教程

### 一、确认新旧站环境

迁移前建议确认：

- 两个网站都已注册 `shop_product` 商品类型。
- 两个网站使用兼容的 Zibll 商城版本。
- 新站已启用商品所需的分类法和购买功能。
- 新站 PHP 支持 ZIP。
- 旧站图片地址可以正常访问。

如果新旧 Zibll 版本使用完全不同的 Meta 字段，数据虽然能迁移，新版主题也可能无法识别旧字段。

### 二、备份

至少备份新站：

- WordPress 数据库
- `wp-content/uploads`
- 已有商品
- Zibll 主题或子主题设置

推荐使用主机快照、WPvivid 或数据库管理工具。

### 三、旧站导出

1. 在旧站安装并启用插件。
2. 打开“工具 → Zibll 商城迁移”。
3. 选择迁移内容：
   - 完整商品 Meta
   - 分类和标签
   - 商品图片
4. 使用状态筛选或搜索框缩小列表。
5. 勾选最终需要迁移的商品。
6. 点击“下载商品迁移包”。
7. 等待页面显示“导出完成”。
8. 保存浏览器下载的 ZIP。

筛选仅控制当前显示结果；最终以每个商品前的复选框是否勾选为准。

### 四、新站导入

1. 在新站安装并启用同一插件。
2. 确认 Zibll 商城环境已经启用。
3. 打开“工具 → Zibll 商城迁移”。
4. 上传旧站生成的 ZIP。
5. 建议勾选：
   - 覆盖之前导入或别名相同的商品
   - 保留原商品状态
6. 点击“开始导入”。
7. 等待自动返回结果页面。

### 五、导入后检查

抽查至少三个不同类型的商品：

1. 普通定价商品。
2. 有规格或多属性的商品。
3. 有图库、正文图片或下载资源的商品。

重点检查：

- 售价和原价
- 库存
- 销量
- 商品规格和属性
- 购买按钮
- 特色图和图库
- 正文图片
- 分类层级
- 私密、草稿或定时状态

## 导入选项说明

### 覆盖已有商品

勾选后会更新之前由本插件导入的商品；如果没有来源标识，则按商品别名查找。

不会仅因为标题相同就跳过。

### 保留原商品状态

勾选后：

- 已发布仍为已发布
- 私密仍为私密
- 草稿仍为草稿
- 待审仍为待审
- 定时商品仍保留定时状态

取消后所有商品统一导入为草稿，适合先检查再发布。

## Meta 迁移规则

默认迁移所有商品 Meta，但排除：

```text
_edit_lock
_edit_last
_thumbnail_id
_wp_old_slug
_wp_trash_meta_status
_wp_trash_meta_time
插件自身来源标识
```

原因是这些字段属于编辑锁、回收站或旧站 ID，直接复制可能破坏新站数据。

序列化数组会安全还原；序列化 PHP 对象会被拒绝并记录警告。

## 图片迁移规则

支持：

- JPEG
- PNG
- GIF
- WebP
- AVIF
- BMP
- ICO
- 安全清理后的 SVG

每张图片最大默认限制为 50MB。

SVG 会移除：

- 脚本
- 事件属性
- 外部资源
- iframe、object、embed
- 动画和危险样式

WordPress 默认不允许 SVG 上传。插件只会在写入当前已经安全清理的迁移 SVG 时临时允许 `image/svg+xml`，处理完成后立即移除放行规则，不会永久开放网站的普通 SVG 上传。

同时兼容部分带有额外前后内容、DOCTYPE 或常见 HTML 实体的非标准 SVG。

相同图片会按 SHA-256 哈希复用。

如果商品写入失败，且某张图片没有被其他成功商品使用，本插件会清理这次新建的孤立媒体。

## 导入报告

结果页分别显示：

- 新增商品
- 覆盖商品
- 跳过商品
- 新增图片
- 失败商品及原因
- 失败图片、关联商品、旧地址及原因
- Meta、分类等其他警告

“商品成功但有警告”表示商品基础记录已写入，但某个 Meta、分类或图片需要人工检查。

## 服务器要求

- WordPress 5.8 或更高版本
- PHP 7.4 或更高版本
- PHP `ZipArchive`
- SVG 需要 PHP DOM 扩展
- 足够的磁盘空间
- 旧站可访问商品图片

大批量商品建议：

```ini
memory_limit = 512M
max_execution_time = 600
upload_max_filesize = 1536M
post_max_size = 1536M
```

部分主机还限制 Web 服务器请求时间，需要联系主机商调整。

## 不迁移的内容

本插件专门迁移商品，不迁移：

- 用户账号
- 订单
- 支付流水
- 用户余额
- 提现记录
- 优惠券使用记录
- 评论和评价
- 整站主题设置
- 商城全局配置

这些数据涉及用户关系和交易完整性，不应通过简单商品克隆处理。

## 安全与隐私

- 只导入由本插件生成且来源可信的 ZIP。
- 商品 Meta 可能包含授权、下载或购买配置。
- 不要把真实商城迁移包上传到公开仓库或公共网盘。
- 公开仓库中只包含插件源码，不包含任何商城数据。
- 导入前备份，导入后抽查。

## 常见问题

### 为什么迁移包很大？

ZIP 会包含所选商品的图片。商品数量、图库数量和原图大小决定迁移包体积。

### 可以分批迁移吗？

可以。使用搜索或状态筛选，取消全选后分批勾选商品即可。

### 可以重复导入吗？

可以。建议勾选覆盖，插件会更新之前导入的商品，并复用相同图片。

### 为什么价格存在但前台不显示？

通常是新旧 Zibll 版本使用不同字段结构，或新站缺少对应商城模块。请比较新旧商品 Meta 和主题版本。

### 为什么某个 Meta 被跳过？

如果该值包含序列化 PHP 对象，为避免对象注入，新版会跳过并报告警告。普通数组、字符串和数字不会受影响。

### 百分比和预计时间如何计算？

导入百分比按实际完成的图片与商品任务合并计算；导出按实际完成的商品数计算。预计时间根据当前平均处理速度动态更新，大图片下载期间可能短暂波动。

达到 100% 后会显示右上角和底部关闭按钮。完成窗口不会自动消失，必须由用户手动关闭。
